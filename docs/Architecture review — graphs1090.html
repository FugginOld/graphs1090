1 · Centralize the RRD metric catalog

Strong in-process

Files

  • scripts/graphs1090.sh — graph DEF: clauses (×23 functions)
  • scripts/graphs1090.sh — stats_cpu() L1387 · write_stats_json() L1407
  • (downstream) collector/adsb_telegraf.py — the new decoder reader

Before — knowledge scattered

Each reader re-states the file:DS string. SDR variant → fix in N places.

After — one catalog

rrd_last name not rrd_last path ds. One row per metric.

Problem
“Metric X = file Y, DS Z” is restated in every graph DEF, in stats_cpu, and in write_stats_json — many readers, no owner.

Solution
One associative array METRIC[name]="file:ds"; rrd_last/stats_cpu/write_stats_json resolve through it.

Wins

  • locality: DS names in one module
  • leverage: one catalog, N readers
  • de-risks the live SDR migration (null-stats fix = 1 line)
  • interface shrinks: rrd_last name
Honest limit: the 23 graph functions stay — their per-graph CDEF/VDEF logic genuinely varies, so they are deep, not shallow. The catalog deepens the readers (stats + future decoder remap), not the renderers.

2 · One source of truth for themes

Strong in-process

Files

  • scripts/graphs1090.sh — set_palette() L44 · write_themes_manifest() L1356
  • html/graphs.js — THEME_META[] L72
  • html/portal.css — [data-theme] blocks L7–12

Before — defined 3× by hand

After — palette is the catalog

Problem
A theme is declared in three hand-synced places; themes.json carries only ids, so JS keeps its own THEME_META and CSS its own blocks. Mismatch silently breaks the switcher.

Solution
set_palette becomes the one catalog (id, name, swatch, chrome, rrd colors); it emits a rich themes.json and a generated themes.css. JS drops THEME_META.

Wins

  • deletion test: THEME_META deletes cleanly
  • locality: add a theme in one module
  • leverage: switcher + chrome + PNG colors from one definition
  • no silent id drift across 3 files
Two-step option: generating CSS is the contentious half. Step 1 (enrich themes.json with name+swatch, delete THEME_META) already kills one hand-synced copy with zero CSS generation. Step 2 (generate themes.css) is optional.

3 · Collapse switchView() into a graph table

Strong in-process

Files

  • html/graphs.js — switchView() L148–299

Before — ~150 selectors, one each

#aircraft-image .attr(src, graphDir+"…"+frame+".png")
#aircraft-link .attr(href, graphDir+"…"+frame+".png")
#tracks-image .attr(src, graphDir+"…"+frame+".png")
#range-image .attr(src, graphDir+"…"+frame+".png")
⋮ ×150
#system-temp-image .attr(src, graphDir+"…")

Same URL pattern retyped per element. Add a graph → hand-edit the god function. Zero JS tests.

After — table + loop + pure builder

GRAPHS = [

  {el:'aircraft', base:'aircraft'},

  {el:'tracks', base:'tracks'}, … ]

GRAPHS.forEach(g => setImg(g, frame))
graphUrl(dir,host,base,frame,ts) // pure → testable

URL generation splits from DOM mutation. Add a graph → one row.

Problem
switchView() is a god function: ~150 jQuery selectors each rebuild the identical URL pattern; no seam between URL building and DOM mutation, so nothing is unit-testable.

Solution
A data table of {el, base} rows + one loop; extract a pure graphUrl(). switchView() just maps the table onto the DOM.

Wins

  • interface shrinks; table absorbs 150 lines
  • first testable JS seam (today: zero)
  • add a graph = one row
  • locality: URL pattern in one place

4 · One pure aircraft-stats module

Worth exploring in-process

Files

  • lib/dump1090.py — compute_aircraft_stats / greatcircle / perc / _quartile_dict L547
  • collector/adsb_stats.py — verbatim copy (this session)

Before — two copies of the math

After — one module, two importers

Problem
The quartile / great-circle / classification math now lives in two identical copies; a fix must land twice (locality lost).

Solution
Keep adsb_stats.py as the one module; have dump1090.py import from it. The collectd plugin's interface does not change.

Wins

  • two adapters justify the seam: plugin + collector
  • locality: fix the math once
  • existing tests already point at the pure module
Tension with the migration decision. The migration plan deliberately chose non-destructive coexistence (“Phase D consolidates”). But the import itself is non-destructive — it changes no interface — so consolidation needn’t wait for Phase D. Counter-argument to weigh: importing across lib/ ↔ collector/ couples the legacy plugin to a new folder it will outlive. Decide before doing.

Top recommendation

1 · Centralize the RRD metric catalog →

Highest leverage and it lands on work already in flight. The Telegraf/InfluxDB migration depends on the exact file:DS names being correct across SDR variants — today that knowledge is restated in stats_cpu, write_stats_json, and every graph DEF. A single METRIC[] catalog turns the predicted “null stats on a different SDR” failure into a one-line fix, and gives the new collector one place to read the same map. Deletion test passes cleanly: remove the catalog and the path strings scatter straight back across every reader.

Pairs naturally with #3 (the JS side gets its first test seam) once the data side has one owner.
