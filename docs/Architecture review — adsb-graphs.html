40 data records replace 40 functions. One seam to test.

Problem

40+ functions are shallow — each wraps one rrdtool graph call with copy-pasted DEF/CDEF/LINE/GPRINT flags; the interface (no params) is nearly as complex as the implementation (manual flag construction).

Solution

One render_graph(spec) module whose interface is a graph spec record; rrdtool invocation, palette resolution, and output path logic move inside. Graph types become data, not functions.

Wins

  • → locality: rrdtool flag bugs concentrate in one module
  • → leverage: one interface, 40 call sites
  • → interface shrinks: add a graph type = add a data record, not a function
  • → tests hit one seam: feed spec, assert PNG exists / rrdtool args
  • → deletion test passes: deleting 39 of 40 functions re-scatters complexity back to callers

2 · Consolidate the aircraft statistics module

Strong in-process

collector/adsb_stats.py · 98 lines  +  lib/dump1090.py · 636 lines (overlapping)

Before — two authorities

After — one authority

Problem

Aircraft filtering, range capping, RSSI clamping, and quartile computation exist in both modules; two modules share one authority. Bug fixes must be applied twice.

Solution

adsb_stats.py becomes the single deep module for all aircraft statistics. dump1090.py (deprecated) is deleted; its callers either already use adsb_stats.py or disappear with it.

Wins

  • → locality: signal/range policy lives in exactly one place
  • → leverage: test suite (386 lines) covers the only copy
  • → delete 636 lines of deprecated implementation (deletion test trivial)
  • → Phase D consolidation noted in SUMMARY.md — this closes it

↓ adapter seam
telegraf_adapter: print(format_line_protocol(metrics))

Problem

17 functions mix transformation with stdout side-effects; the interface is implicit and testing requires capturing stdout rather than crossing a stable seam.

Solution

Split at a seam: collect(json) → list[Metric] is a pure deep module; a thin Telegraf adapter calls it and emits line protocol. Tests cross the collect seam — no stdout capture needed.

Wins

  • → leverage: pure collect() reusable for Prometheus, MQTT, or any adapter
  • → locality: metric computation errors concentrate in collect(), not across 17 formatting functions
  • → tests assert on list[Metric], not parsed stdout strings
  • → two adapters justify the seam; one today makes this speculative until a second materialises
Existing test suite covers adsb_telegraf.py well (386 lines). This deepening improves ergonomics but doesn't fix a coverage gap — weigh cost against benefit.

4 · Extract theme configuration from the graph module

Worth exploring local-substitutable

scripts/adsb-graphs.sh (set_palette)  +  html/portal.css ([data-theme])  +  html/graphs.js (theme switcher)

Before — theme split across 3 modules

After — themes.json is the authority

Problem

Adding a theme requires coordinated edits to three modules. No single seam owns the theme list; a mismatch silently breaks the UI.

Solution

themes.json (already generated per SUMMARY.md) becomes the authority. Three consumers read it; one edit propagates to all three.

Wins

  • → locality: theme-list changes require one edit, not three
  • → leverage: seam already exists — themes.json is already generated
  • → mismatch between generator and UI detectable at load time
SUMMARY.md records themes.json is already written. Verify its schema before treating it as the authority — it may already carry the theme list.

Top recommendation

→ Candidate 1: Collapse the graph generation monolith

adsb-graphs.sh is 1,534 lines of copy-pasted rrdtool invocations. The deletion test is decisive: delete any 39 of the 40 graph functions and their complexity moves back to callers — proving none earn their keep as separate modules. A single render_graph(spec) module absorbs all 40 implementations behind one seam, concentrates rrdtool flag logic in one place (locality), and makes adding a graph type a data edit rather than a function copy (leverage).

Candidate 2 (statistics consolidation) is a close second and lower risk — it mostly deletes code. Tackle both in the same pass.
